# 拖拽排序修复设计

## 背景

应用已为历史记录和导航配置项实现 HTML5 Drag & Drop，但 Windows Tauri 窗口仍启用默认原生拖放处理器。Tauri 明确要求在 Windows 上关闭该处理器后，前端 HTML5 拖放才能工作。此外，顶部导航页签当前只有点击筛选能力，拖放仅存在于导航配置弹窗中。

## 目标

- 恢复全部历史记录和收藏筛选中的拖拽排序。
- 支持直接拖动顶部导航页签调整顺序。
- 保持“全部”页签固定在首位且不可拖动。
- 保留现有排序持久化方式，不改数据库结构和设置结构。

## 非目标

- 不引入第三方拖拽库。
- 不重写虚拟列表，也不增加跨视口自动滚动。
- 不改变置顶记录优先于普通记录的现有排序规则。
- 不调整筛选状态下可见子集排序对全局顺序的现有影响。

## 方案比较

### 方案 A：仅修复 Tauri 配置

改动最小，但只能恢复历史记录和导航配置弹窗的拖放，顶部导航页签仍无法直接拖动，不能完整满足需求。

### 方案 B：修复 Tauri 配置并补齐顶部页签拖放（采用）

关闭 Tauri 原生拖放处理器，继续使用现有 HTML5 Drag & Drop；抽取纯函数处理导航顺序，使顶部页签和配置弹窗共用同一排序规则。改动集中、依赖少，并可通过单元测试覆盖关键排序约束。

### 方案 C：使用 Pointer Events 重写拖拽

可以绕开原生 HTML5 拖放差异，但需要自行实现命中、拖动预览、释放和滚动处理，改动与风险明显超过当前需求。

## 详细设计

### Tauri 窗口配置

在主窗口配置中设置 `dragDropEnabled: false`。这会关闭 Tauri 的原生文件拖放处理器，使 Windows WebView2 将 HTML5 `dragstart`、`dragover` 和 `drop` 事件交给 React 页面。

### 历史记录和收藏

保留现有历史项 `draggable` 及事件处理。收藏筛选返回的记录仍通过同一个历史列表渲染，因此配置修复会同时恢复两者。排序完成后继续调用现有 `reorder_items` 命令持久化。

### 顶部导航页签

- “全部”页签保持不可拖动。
- 其他可见页签绑定与配置弹窗一致的拖放事件。
- 拖动到目标页签时，更新 `nav_filters_config.visible` 并调用现有设置持久化接口。
- 导航排序使用一个纯函数，统一处理无效 key、同项拖放和“全部”固定首位规则，顶部页签与配置弹窗共用。
- 拖动只改变顺序，不切换当前激活筛选。

### 错误处理

界面先进行乐观更新；设置保存失败时记录现有错误日志，不新增弹窗。历史记录排序继续沿用现有失败后刷新数据的行为。

## 测试与验收

1. 配置测试确认主窗口显式设置 `dragDropEnabled: false`。
2. 纯函数测试覆盖：正常移动、同项拖放、无效 key、“全部”始终保持首位。
3. 现有全部前端单元测试通过。
4. TypeScript 类型检查和 Vite 生产构建通过。
5. Windows Tauri 手动验收：
   - “全部”和“收藏”列表均能拖动记录并保持排序。
   - 顶部非“全部”页签可直接拖动，重启后顺序保持。
   - 配置弹窗内的导航排序仍正常。
   - 点击记录仍执行粘贴，点击页签仍执行筛选。
